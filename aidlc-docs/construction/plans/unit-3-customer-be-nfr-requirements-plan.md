## Questions

### 성능

**Q1. 주문 생성 API 응답 시간 목표**
고객이 주문 확정 버튼을 누른 후 응답까지 허용 가능한 시간:

A) 1초 이내 (빠른 응답 우선)
B) 3초 이내 (일반적 수준)
C) 특별한 목표 없음 (합리적 수준이면 됨)

[Answer]: A

**Q2. 동시 주문 처리**
같은 매장에서 여러 테이블이 동시에 주문할 때의 처리 방식:

A) 별도 처리 불필요 — 매장 규모(수십 테이블)에서 동시성 이슈 거의 없음
B) 주문 번호 채번에 대해 비관적 잠금(DB lock) 적용

[Answer]: DB Lock을 사용하고 싶지는 않은데, 동시성 이슈는 언제나 있지. 다른 옵션을 제안해줘. 

### 안정성

**Q3. 주문 생성 실패 시 트랜잭션 처리**
세션 생성 + 주문 생성이 한 트랜잭션인데, 부분 실패 시:

A) 전체 롤백 (세션도 주문도 생성 안 됨) — 단순하고 안전
B) 세션은 유지하고 주문만 롤백 — 세션 재생성 방지

[Answer]: A

**Q4. SSE 이벤트 발행 실패 시**
주문은 성공했는데 SSE 이벤트 발행이 실패한 경우:

A) 주문 성공 유지 — SSE 실패는 무시 (관리자가 새로고침으로 확인 가능)
B) 주문도 롤백 — SSE 발행까지 성공해야 주문 완료

[Answer]: A로 하되, 관리자 화면에서 주기적으로 refresh가 일어나야 주문이 누락되지 않을 것 같아. 이건 옆 팀의 담당 업무니까 거기서 알 수 있도록 해줘.
